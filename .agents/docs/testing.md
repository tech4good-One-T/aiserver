# 테스트 코드 작성 지침

## 기본 원칙

- 모든 동작 변경에는 성공 경로와 중요한 실패 경로를 검증하는 테스트를 추가한다.
- 테스트는 구현 세부사항보다 외부에서 관찰 가능한 입력과 결과를 검증한다.
- 하나의 테스트는 하나의 동작만 설명하며 `test_<조건>_<기대결과>` 형태로 이름을 짓는다.
- 테스트 간에 실행 순서나 변경 가능한 전역 상태를 공유하지 않는다.
- 시간, 네트워크, 외부 API, 데이터베이스는 경계에서 대체하고 실제 외부 서비스에 의존하지 않는다.
- 버그 수정 시에는 수정 전 실패하고 수정 후 통과하는 회귀 테스트를 먼저 추가한다.

## FastAPI 테스트

- API 동작은 `fastapi.testclient.TestClient`로 상태 코드와 응답 본문을 함께 검증한다.
- 수명 주기 이벤트가 실행되도록 `TestClient`를 컨텍스트 관리자로 사용한다.
- 검증 오류, 인증 실패, 리소스 미존재 등 계약에 포함된 오류 응답도 테스트한다.
- 의존성 대체가 필요하면 `app.dependency_overrides`를 사용하고 테스트 종료 시 반드시 비운다.
- 비동기 함수를 직접 검증해야 할 때만 비동기 테스트를 사용한다. HTTP 계약 테스트는 우선
  `TestClient`로 작성한다.

## 구조와 픽스처

- 테스트 파일은 `tests/test_<대상>.py`에 둔다.
- 여러 파일에서 재사용하는 픽스처만 `tests/conftest.py`에 둔다.
- Arrange, Act, Assert 단계를 빈 줄로 구분해 의도를 드러낸다.
- 테스트 데이터는 필요한 최소 필드만 명시하고 임의 값 생성에 기대지 않는다.

## 실행 기준

PR을 올리기 전에 다음 검사를 모두 통과시킨다.

```bash
uv run ruff check .
uv run ruff format --check .
uv run pytest
```
