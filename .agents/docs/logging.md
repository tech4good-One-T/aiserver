# 로깅 지침

## 목적

운영 중 문제를 재현할 수 있도록 애플리케이션 이벤트를 일관된 형식과 수준으로 남긴다.
기본 설정은 `app/core/logging.py`에서 한 번만 관리한다.

## 작성 규칙

- 모듈마다 `logger = logging.getLogger(__name__)`를 선언한다.
- `print` 대신 표준 `logging`을 사용한다.
- 메시지는 검색 가능한 사건 중심 문장으로 작성하고, 값은 지연 치환 방식으로 전달한다.
  예: `logger.info("Job completed (job_id=%s)", job_id)`
- 정상적인 주요 상태 변화는 `INFO`, 복구 가능한 이상은 `WARNING`, 요청 또는 작업 실패는
  `ERROR`, 개발용 상세 정보는 `DEBUG`를 사용한다.
- 예외를 기록할 때는 `logger.exception(...)`으로 스택 트레이스를 포함한다.
- 비밀번호, 토큰, 인증 헤더, 개인정보 원문은 어떤 로그 수준에서도 기록하지 않는다.
- 동일한 예외를 여러 계층에서 반복 기록하지 않는다. 예외를 처리하거나 외부 응답으로 변환하는
  경계에서 한 번 기록한다.
- 요청별 추적 ID가 도입되면 모든 요청 로그에 같은 필드명 `request_id`를 사용한다.

## 설정과 실행

- 로그 수준은 `LOG_LEVEL` 환경변수로 설정하며 기본값은 `INFO`다.
- 허용 값은 `DEBUG`, `INFO`, `WARNING`, `ERROR`, `CRITICAL`이다. 잘못된 값은 시작 시
  오류로 처리해 설정 실수를 숨기지 않는다.
- 로깅 설정은 애플리케이션 진입점에서 한 번 호출하고, 개별 모듈에서 핸들러를 추가하지 않는다.
- 운영 환경의 파일 보관과 전송은 프로세스 밖의 실행 플랫폼에서 처리한다. 애플리케이션은
  표준 출력으로 기록한다.

## 검증

로그 수준이나 포맷을 변경할 때는 `tests/test_logging.py`를 함께 수정하고 아래 명령을 실행한다.

```bash
uv run pytest tests/test_logging.py
```
